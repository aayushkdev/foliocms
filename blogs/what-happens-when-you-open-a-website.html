What Actually Happens When You Open a Website

March 2026 · 7 min read

When you type a website like example.com into your browser and press enter, a surprising number of systems work together before the page appears.

From DNS lookups to encrypted connections and HTTP requests, a single page load involves multiple network protocols and infrastructure layers.

Step 1: DNS Resolution

The browser first needs to know the server’s IP address. Domain names are human-friendly labels, but computers communicate using IP addresses.

Your system queries a DNS resolver which may contact several servers:

  • Root DNS servers
  • TLD servers (.com, .org)
  • The authoritative server for the domain

Eventually the resolver returns something like: 

example.com → 93.184.216.34

Step 2: TCP Handshake

Before sending HTTP data, the browser must establish a reliable connection. TCP performs a three-way handshake: 

SYN → 
SYN-ACK ←
ACK →

After this exchange, both sides agree to start communication.

Step 3: TLS Encryption

If the site uses HTTPS (which most modern websites do), the browser negotiates a TLS session.

This process:

  • Verifies the server certificate
  • Negotiates encryption algorithms
  • Generates session keys

Once complete, communication becomes encrypted.

Step 4: HTTP Request

The browser finally sends the actual request: 

GET / HTTP/1.1
Host: example.com
User-Agent: browser

The server processes the request and returns a response containing HTML, headers, and possibly cookies.

Step 5: Rendering the Page

Once the browser receives the HTML, it begins rendering:

  • HTML is parsed into a DOM
  • CSS is applied
  • JavaScript executes
  • Additional resources load

Only after all these steps do you actually see the page.

Final Thoughts

Opening a simple webpage involves DNS infrastructure, TCP networking, encryption protocols, HTTP servers, and browser rendering engines.

Understanding these layers helps explain why performance, latency, and caching are so important on the modern web.

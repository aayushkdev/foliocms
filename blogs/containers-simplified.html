Containers Simplified

March 2026 · 6 min read

Containers power almost everything in modern cloud infrastructure. Docker, Kubernetes, and serverless platforms all rely on them. But despite how common containers are, many developers still don't fully understand what they actually are.

This post explains containers from first principles and how they work internally.

What is a Container?

A container is simply a process running on your system with isolated resources.

Unlike virtual machines, containers do not emulate hardware or run their own kernel. They share the host operating system but operate in a restricted environment.

This isolation is achieved through two key Linux features:

  • Namespaces
  • Control Groups (cgroups)

Namespaces: Process Isolation

Namespaces allow a process to see a different version of system resources such as processes, filesystems, and networks.

For example, inside a container, a process may think it is running as PID 1 even though the host system has thousands of processes.

Linux provides several namespace types:

  • pid – process isolation
  • net – network interfaces
  • mnt – filesystem mounts
  • uts – hostname isolation
  • ipc – inter-process communication

Cgroups: Resource Limits

While namespaces isolate resources, cgroups control how much of those resources a container can use.

They allow the system to enforce limits like:

  • CPU usage
  • memory usage
  • disk IO

This prevents a container from consuming all system resources.

How Docker Actually Works

Docker is essentially a tool that orchestrates these kernel features.

When you run: 

docker run nginx

Docker performs several steps:

  • Creates a new filesystem layer
  • Starts a process in isolated namespaces
  • Applies resource limits using cgroups
  • Connects networking

At the end of the day, a container is still just a normal process running on your machine.

Containers vs Virtual Machines

The biggest difference is the kernel.

Virtual machines run a full guest OS and kernel, while containers share the host kernel.

This makes containers:

  • lighter
  • faster to start
  • more efficient

Why I Built My Own Container Runtime

While working on my project Crate, I wanted to understand containers beyond Docker abstractions.

Building a runtime from scratch helped me learn how namespaces, filesystem isolation, and process control actually interact inside the Linux kernel.

It also made me appreciate how much complexity Docker hides from developers.

Final Thoughts

Containers are often treated as a magical technology, but under the hood they are simply Linux processes with isolation features.

Understanding how they work makes it easier to debug production systems, design better infrastructure, and appreciate the tools we use every day.

The next time you run docker run, remember that you're not launching a virtual machine — you're just starting a very well isolated process.
